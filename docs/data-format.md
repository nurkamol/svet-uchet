# Формат выгрузки EX518 / EX18

Суточные снимки всех регистров в 00:00, одна строка = один день.

Формат задаёт **программа считывания**, а не сам счётчик, поэтому однофазный EX18 и
трёхфазный EX518 (один вендор, одни интерфейсы PLC/RS-485/оптопорт, одни 4 зоны)
выгружаются одинаково. Приложению число фаз неизвестно и не нужно: из строки читаются
только дата и накопительное `+A`. У однофазного не бывает статуса
`[Phase Sequence Reversal]` и нет пофазных колонок — ни то, ни другое на расчёт не влияет.

| Особенность | Детали |
|---|---|
| Показания | Накопительные (одометр). Расход дня X = показание(X+1) − показание(X) |
| Граница месяца | Показание на 1-е число 00:00 = итог предыдущего месяца |
| Даты | Строки `dd/mm/yyyy hh:mm:ss`; возможны серийные даты Excel |
| Числа | Десятичный разделитель — запятая (`47,53`) |
| Пустой статус | Литеральный текст `null` (не пустая ячейка) |
| Ключевые колонки | `Clock` (дата), `Active energy import (+A)(kWh)` (для оплаты), `Meter internal status` |
| Т1–Т4 | Тарифные тайм-зоны; сумма = общий +A; при блочном тарифе не используются |
| Реактивная (+R/−R) | Населением не оплачивается |
| Известные статусы | `[Phase Sequence Reversal]` — обратный порядок фаз, на учёт не влияет |

## Ловушка: SheetJS и .csv

`.csv` нужно читать **`XLSX.read(data, {raw:true})`**, иначе SheetJS типизирует ячейки
сам и молча портит выгрузку:

| В файле | Без `raw:true` | Верно |
|---|---|---|
| `108,59` | `10859` (запятая = разделитель тысяч) | `108.59` |
| `02/01/2026` | серийная `46054` = 1 февраля (прочитано как mm/dd) | 2 января |

С `raw:true` ячейки остаются строками, а их разбирают `parseNum` (знает запятую) и
`parseDate` (знает dd/mm и серийные даты, в том числе пришедшие строкой).

## Незнакомые выгрузки

Если `findCols` не опознал заголовки, приложение **не гадает**: показывает первые строки
файла и просит указать колонки вручную (`rowsFromCols`). Так подходит выгрузка любого
счётчика, а не только EX518/EX18 — и без единого предположения о формате.

Вторая страховка — `implausibleDays`: расход выше `MAX_PLAUSIBLE_KWH_PER_DAY` (500 кВт·ч
в сутки) для бытового счётчика физически невозможен — EX18 при 60 A × 220 В отдаёт максимум
~317. Такой день означает, что файл прочитан неверно, и пользователь видит предупреждение.
Именно эта проверка поймала бы баг с запятой (он давал ~5200 кВт·ч в сутки).
На `.xlsx` опция не влияет: там даты и числа уже типизированы — это проверено тестами
`.xlsx: …` в `core.test.js`. Тесты по `demo.csv` проверяют конкретные значения
(январь = 307,2 кВт·ч → 226 480 сум), а не только «расход ≥ 0», — иначе искажение
такого рода снова пройдёт незамеченным.
